## BLOQUE 1 -- Análisis de código inseguro

Analiza el siguiente código Java:

/\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*/

``` java
import java.sql.*;
import java.util.Scanner;

public class Login {

    public static void main(String[] args) throws Exception {

        Scanner sc = new Scanner(System.in);

        System.out.println("Usuario:");
        String user = sc.nextLine();

        System.out.println("Password:");
        String pass = sc.nextLine();

        Connection con = DriverManager.getConnection(
                "jdbc:mysql://localhost/app", "root", "root");

        Statement st = con.createStatement();

        String query = "SELECT * FROM usuarios WHERE nombre='"
                + user + "' AND password='" + pass + "'";

        ResultSet rs = st.executeQuery(query);

        if(rs.next()) {
            System.out.println("Acceso concedido");
        } else {
            System.out.println("Acceso denegado");
        }

        con.close();
    }
}
```

/\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*\*/

### Pregunta 1.1

Identifica al menos 4 vulnerabilidades o malas prácticas de seguridad en
el código.

Explica brevemente cada una.

1. SQL Injection

String query = "SELECT * FROM usuarios WHERE nombre='"
    + user + "' AND password='" + pass + "'";
    
En esta query el usuario si introduce como username 1 = 1 podrá ganar acceso como administrador al sistema.

Esto ocurre porque 1 es siempre igual a 1, por lo tanto, siempre es correcto y por ello el login será exitoso.



2. Exposición de Enpoint
   "jdbc:mysql://localhost/app"

Puede ser mala práctica exponer el endpoint donde está la base de datos en el código.

3. Credenciales en código
    "root", "root"

 No se deben incluir las credenciales de inicio de sesión como root directamente en el código porque si una persona no deseada consigue acceso al código ya tendría acceso a la base de datos  ya que en el anterior mala práctica viene la ruta para entrar en ella.

4. Dos/Buffer Overflow... 

No se controla el límite de la cadena que introduce el usuario

## BLOQUE 2 -- Conceptos de programación segura

Responde a las siguientes preguntas:

### Pregunta 2.1

Explica qué es una vulnerabilidad de tipo SQL Injection y cómo se
previene en Java.

Es una vulnerabilidad que permite al usuario ejecutar consultas SQL directamente sobre la base de datos.

Se previene sanitizando todo lo que interactue con la base de datos, no solo los formularios.

En código se previene usando lógica para separar lo que introduce el usuario a las operaciones que se hacen en el backend.

### Pregunta 2.3

¿Qué es el principio de mínimo privilegio? Pon un ejemplo en Java o en
bases de datos.

Es un principio orientado al control de acceso y las vulnerabilidades relacionadas con él. Como las de la categoría Improper Acces Control o Broken Acces Control.

Este principio se usa para que los usuarios que pueden acceder a una aplicación tengan los mínimos privilegios posibles y así limitar en las medida de los posible las acciones que el usuario puede hacer. Es útil para evitar que un usuario corriente ejecute aciones que solo las debería de poder ejecutar un administrador

### Pregunta 2.4

Indica si las siguientes afirmaciones son verdaderas o falsas y
justifica:

a)  PreparedStatement evita SQL Injection\

Falsa

Depende de la implementación de la consulta preparada. Si es una consulta preparada que directamente hace algo ya definido como obtener el precio de todos los productos que contengan la letra "a" y no puede ser manipulada la consulta no habría SQL Injection.


Pero si es una consulta preparada que depende de x e y y esas variables se obtienen de los datos que introduce el usuario puede haber SQL Inyection




b)  RSA es criptografía simétrica\



Falsa. RSA es criptografía asimétrica, es decir se firma con la clave pública y con la clave privada se desifra



c)  La clave privada puede compartirse públicamente\

Falsa. La clave privada nunca debe compartirse ya que si se comparte alguien podría usarla para cifrar mensajes haciendose pasar por ti o descifrar mensajes que son para ti 


d)  Un certificado digital contiene una clave pública

Sí, contiene una clave pública. Se utiliza tanto una clave pública como una privada
